# منهجية تتبع الأخطاء وتحليلها — Claude Code

> دليل شامل يشرح الخوارزميات والأساليب المستخدمة في اكتشاف الأخطاء وتشخيصها وإيجاد الحلول في مشروع SAJCO وأي مشروع مشابه.

---

## المرحلة الأولى: قراءة الخطأ وتصنيفه

### خوارزمية التصنيف الأولي

```
إذا جاء الخطأ من المستخدم:
  ├── هل يحمل رسالة خطأ واضحة (console error, 404, 500)؟
  │     نعم → انتقل لـ [تحليل رسالة الخطأ]
  │     لا  → انتقل لـ [وصف الأعراض]
  │
  └── هل الخطأ وظيفي (شيء لا يعمل) أم بصري (شيء لا يظهر)؟
        وظيفي → ابحث في منطق الكود (JS functions, data flow)
        بصري  → ابحث في CSS, DOM, conditions
```

### تصنيف الأخطاء في هذا المشروع

| النوع | الأعراض الشائعة | مكان البحث الأول |
|-------|----------------|-----------------|
| **Supabase Sync** | البيانات تختفي بعد reload | `_syncToSupabase()`, `_loadRemoteDB()` |
| **RLS / Auth** | 401, 403, "violates row-level security" | Supabase Dashboard → Policies |
| **Race Condition** | يعمل أحياناً ولا يعمل أحياناً | `async/await`, event handlers |
| **UUID Mismatch** | فلترة لا تُعطي نتائج | مقارنة IDs بين localStorage و Supabase |
| **FK Constraint** | 409 Conflict, foreign key violation | ترتيب عمليات الإدراج في Supabase |
| **Duplicate Definition** | دالة تعمل بشكل مختلف عن المتوقع | `grep` اسم الدالة في كل الملفات |
| **Stale Data** | بيانات قديمة تظهر بعد تعديل | `await _loadRemoteDB()` قبل العرض |

## المرحلة الثانية: تحليل رسالة الخطأ

### خوارزمية تفسير رسائل الـ Console

```
رسالة الخطأ
    │
    ├── تحتوي على "undefined" أو "null"
    │     → متغير لم يُعرَّف أو بيانات لم تُحمَّل بعد
    │     → ابحث عن await مفقود قبل _loadRemoteDB()
    │
    ├── تحتوي على "is not a function"
    │     → الدالة مُعرَّفة في مكان آخر أو اسمها مختلف
    │     → grep للاسم في جميع الملفات
    │
    ├── تحتوي على "foreign key constraint"
    │     → تحاول إدراج سجل يُشير لـ ID غير موجود
    │     → تأكد أن الجداول الأصل تُحفَظ أولاً
    │
    ├── تحتوي على "row-level security"
    │     → المستخدم لا يملك صلاحية الكتابة
    │     → تحقق من RLS policies في Supabase
    │
    ├── تحتوي على "400 Bad Request"
    │     → البيانات المُرسَلة للـ API بصيغة خاطئة
    │     → تحقق من أنواع البيانات (UUID vs text, null vs undefined)
    │
    ├── تحتوي على "409 Conflict"
    │     → تعارض unique constraint
    │     → إما بيانات مكررة أو FK مفقود
    │
    └── تحتوي على "Maximum call stack"
          → استدعاء دالة لنفسها (recursion لا نهائي)
          → ابحث عن دالة تستدعي نفسها باسم مختلف أو نفس الاسم
```

## المرحلة الثالثة: تتبع مسار البيانات (Data Flow Tracing)

هذه أقوى تقنية في التشخيص. الهدف: تتبع البيانات من المصدر إلى العرض.

### خوارزمية تتبع مسار البيانات

```
[المستخدم يضغط زر] 
        ↓
[دالة onclick تُستدعى]
        ↓
[تُعدِّل db محلياً]
        ↓
[saveDB() → _syncToSupabase()]
        ↓
[_loadRemoteDB() عند إعادة التحميل]
        ↓
[دالة render تقرأ db وتعرض]
        ↓
[المستخدم يرى النتيجة]
```

**عند وجود خطأ:** أسأل في أي خطوة توقف المسار؟

```
فحص كل خطوة:
1. هل الدالة onclick تُستدعى؟           → console.log في بدايتها
2. هل db تحتوي على البيانات الصحيحة؟   → console.log(db.tableName)
3. هل saveDB() تُكمل بدون خطأ؟          → تحقق من console errors
4. هل Supabase يقبل البيانات؟           → تحقق من Network tab
5. هل _loadRemoteDB() يُحضر البيانات؟   → تحقق من الاستجابة
6. هل دالة render تُشغَّل بعد التحميل؟  → تحقق من ترتيب الاستدعاءات
```

## المرحلة الرابعة: خوارزميات الكشف عن أنماط الأخطاء الشائعة

### 1. خوارزمية كشف Race Condition

```
هل الخطأ يحدث بشكل متقطع (أحياناً يعمل، أحياناً لا)؟
        ↓ نعم
هل هناك async functions تُستدعى من event handler؟
        ↓ نعم
هل الاستدعاءات متزامنة (بدون await أو debounce)؟
        ↓ نعم
→ Race Condition مؤكدة

الحل:
  أ) إضافة debounce إذا كان المصدر input events
  ب) إضافة رقم تسلسلي (seq) لإلغاء الردود القديمة
  ج) استخدام Promise chain بدلاً من Promise.all إذا يوجد تبعية
```

**مثال تطبيقي من المشروع:**
```javascript
// المشكلة: كل ضغطة مفتاح تُرسل طلب Supabase
input.addEventListener('input', () => checkEmpIdAllowed(val));

// الحل: debounce + sequence number
let seq = 0;
input.addEventListener('input', () => {
    clearTimeout(timer);
    timer = setTimeout(() => {
        const mySeq = ++seq;
        checkEmpId(val).then(result => {
            if (mySeq !== seq) return; // تجاهل الرد القديم
            // استخدم النتيجة
        });
    }, 350);
});
```

### 2. خوارزمية كشف مشاكل UUID / ID Mismatch

```
هل الفلترة لا تُعطي نتائج رغم وجود بيانات؟
        ↓ نعم
هل الفلترة تستخدم المقارنة بالـ ID مباشرة؟
        ↓ نعم
هل البيانات تأتي من مصدرين مختلفين (localStorage + Supabase)؟
        ↓ نعم
→ UUID Mismatch محتمل

التشخيص:
  console.log(currentUser.id)         // ID من الجلسة
  console.log(db.users[0].id)         // ID من Supabase
  هل هما متطابقان؟

الحل:
  بدلاً من: filter(c => c.userId === currentUser.id)
  استخدم:  filter(c => {
               const u = db.users.find(x => x.id === c.userId);
               return u && u.empId === currentUser.empId; // empId لا يتغير
           })
```

### 3. خوارزمية كشف Duplicate Function Definition

```
دالة تتصرف بشكل مختلف عن المتوقع؟
        ↓
ابحث عن اسمها في كل الملفات:
  grep -rn "function functionName" src/

هل يوجد أكثر من تعريف؟
        ↓ نعم
→ آخر تعريف يُلغي السابق في JavaScript

الحل:
  - احذف التعريف المكرر أو دمجهما في واحد
  - إذا كانا مختلفَين، أعد تسمية أحدهما
```

**مثال من المشروع:**
```javascript
// التعريف الأول (صحيح، مع destination parameter)
function returnFromAgent(id, newCalDate, destination) { ... }

// التعريف الثاني (بسيط، يُلغي الأول!)
function returnFromAgent(id, newCalDate) { ... }
// النتيجة: الأول لا يعمل أبداً
```

### 4. خوارزمية تشخيص مشاكل Supabase Sync

```
البيانات تختفي بعد reload؟
        ↓
هل saveDB() تُستدعى بعد التعديل؟
        ↓ نعم
هل _syncToSupabase() تُنفَّذ بدون خطأ؟
        ↓ تحقق من Network tab

إذا وجد خطأ 401:
  → جلسة Supabase منتهية أو المستخدم بدون auth_uid
  → الحل: تجديد الجلسة قبل الطلب

إذا وجد خطأ 403 / RLS violation:
  → السياسة تمنع هذا المستخدم
  → الحل: تعديل RLS policy في Supabase Dashboard

إذا وجد خطأ 409 / FK constraint:
  → جدول يُشير لجدول آخر لم يُحفَظ بعد
  → الحل: مزامنة على مراحل (phase1: parents, phase2: children)

إذا لا يوجد خطأ لكن البيانات لا تُحفَظ:
  → الدالة تُحدِّث db محلياً فقط بدون Supabase call
  → الحل: إضافة supabaseClient.from(...).update/insert
```

### 5. خوارزمية تحليل Notification Flow

```
للتحقق من اكتمال نظام الإشعارات:

لكل عملية مهمة:
  1. من يُنفِّذ العملية؟ (admin / head / surveyor)
  2. من يُتأثَّر بها؟ (owner / head / admin / requester)
  3. هل كل طرف متأثر يُبلَّغ؟

اصنع مصفوفة:
  ┌─────────────────┬───────┬──────┬─────────┬───────────┐
  │ الحدث           │ Admin │ Head │ Surveyor│ Requester │
  ├─────────────────┼───────┼──────┼─────────┼───────────┤
  │ طلب صيانة      │  ✓    │  ✓   │    ✓    │     —     │
  │ موافقة على طلب │  —    │  ✓   │    ✓    │     ✓     │
  │ رفض طلب        │  —    │  ✓   │    ✓    │     ✓     │
  └─────────────────┴───────┴──────┴─────────┴───────────┘

الفراغات (—) = إشعارات مفقودة → أضفها
```

### 6. خوارزمية تتبع Status Flow

```
لكل كيان له حالات (devices, custodies, requests):

ارسم مخطط الحالات:
  warehouse → assigned → maintenance → at_maintenance → warehouse/assigned
                       ↘ needs_calibration → at_calibration → warehouse/assigned

لكل انتقال:
  1. أي دالة تُنفِّذه؟
  2. هل تُحدِّث Supabase؟
  3. هل ترسل الإشعارات الصحيحة؟
  4. ماذا يحدث إذا فُشِل الانتقال؟ هل تُعاد الحالة السابقة؟

الفخ الشائع:
  - حالة تتغير محلياً لكن لا تُزامَن مع Supabase
  - حالة تبقى في "intermediate state" إذا رُفِض الطلب
```

## المرحلة الخامسة: استراتيجية البحث في الكود

### ترتيب البحث من الأسرع للأعمق

```
1. grep للكلمة المفتاحية في الملف المشكوك فيه
   → سريع جداً، يُعطي فكرة فورية

2. قراءة الدالة كاملة بالسياق
   → Read tool مع offset لفهم المنطق الكامل

3. تتبع سلسلة الاستدعاءات (Call Chain)
   → من أين تُستدعى هذه الدالة؟
   → grep "functionName(" في كل الملفات

4. فحص جميع الملفات المرتبطة
   → Explore agent للتحليل الشامل

5. مقارنة السلوك المتوقع مع الفعلي
   → قراءة الـ HTML (UI) + الـ JS (logic) معاً
```

### أولويات الملفات في هذا المشروع

```
app-core.js         → جميع منطق البيانات والدوال المشتركة
supabase-data-layer.js → الاتصال بـ Supabase وتحميل البيانات
[page].html         → منطق الصفحة والـ UI
en/[page].html      → النسخة الإنجليزية (نفس المنطق)
excel-export.js     → تصدير البيانات
```

## المرحلة السادسة: تصميم الحل

### خوارزمية اختيار الحل الأمثل

```
عند وجود حل:
        ↓
هل الحل يُؤثِّر على بيانات موجودة؟
        ↓ نعم → اختبر على بيانات وهمية أولاً
        ↓ لا  → طبِّق مباشرة

هل الحل يغيّر سلوكاً موجوداً يعتمد عليه كود آخر؟
        ↓ نعم → ابحث عن كل مكان يستخدم هذا السلوك (grep)
        ↓ لا  → طبِّق مع اختبار

هل الحل يتطلب migration في Supabase؟
        ↓ نعم → استخدم apply_migration مع اسم وصفي
        ↓ لا  → عدِّل الكود فقط

هل الحل يؤثر على ملفات متعددة؟
        ↓ نعم → طبِّق على الـ AR + EN معاً في نفس الـ commit
```

### مبادئ تصميم الحل

1. **الحد الأدنى من التغيير** — لا تُعدِّل ما لا يحتاج تعديل
2. **الجذر لا الأعراض** — اعالج سبب المشكلة لا مجرد إخفائها
3. **لا تكسر الموجود** — تأكد أن الإصلاح لا يخلق مشكلة جديدة
4. **مزامنة فورية** — أي تعديل يجب أن يذهب لـ Supabase مباشرة
5. **إشعار المتأثرين** — كل تغيير في حالة بيانات = إشعار للأطراف المعنية

## المرحلة السابعة: التحقق بعد الإصلاح

### قائمة فحص ما بعد الإصلاح

```
بعد كل إصلاح:
  □ هل الخطأ الأصلي اختفى؟

  □ هل توجد دوال أخرى تعاني من نفس المشكلة؟
  □ هل الإصلاح يكسر أي وظيفة أخرى؟
  □ هل يوجد شيء في BUGS_AND_FIXES.md يجب تحديثه؟
```

## ملخص الخوارزميات حسب نوع الخطأ

| الخطأ | الخوارزمية | الوقت المتوقع |
|-------|-----------|--------------|
| شيء لا يظهر | تتبع مسار البيانات + فحص async/await | 5-10 دقائق |
| بيانات تختفي بعد reload | فحص Supabase sync + RLS | 10-20 دقيقة |
| يعمل أحياناً فقط | خوارزمية Race Condition | 15-30 دقيقة |
| فلترة لا تُعطي نتائج | خوارزمية UUID Mismatch | 5-10 دقائق |
| دالة تتصرف بشكل غريب | Duplicate Definition check | 2-5 دقائق |
| إشعارات ناقصة | Notification Matrix | 10-15 دقيقة |
| تحليل شامل لميزة | Explore Agent + مصفوفة الحالات | 20-40 دقيقة |
